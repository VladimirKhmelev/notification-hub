# Roadmap: 4 спринта по 2 недели

Каждая задача — отдельная единица работы. После выполнения задачи агент
должен: показать, что изменилось, убедиться что `make up` работает,
закоммитить, и остановиться в ожидании подтверждения перед следующей задачей.

Все спринты 1–3 — только localhost. Домен/TLS/VPS — не раньше Спринта 4,
Задача 4.1.

---

## Спринт 1: сквозной скелет, один источник, UI уже есть

**Задача 1.1 — каркас репозитория**
Инициализировать Go-модуль, структуру папок (`cmd/`, `internal/`, `web/`),
`docker-compose.yml` с Postgres, `Makefile` с целями `up`, `down`, `logs`.
Пустой health-check эндпоинт в `api`, поднимается и отвечает 200.

**Задача 1.2 — схема БД и миграции**
Таблицы `sources` (id, type, name, config jsonb, created_at) и
`events` (id, source_id, title, body, priority, created_at, read_at).
Миграции любым простым инструментом (`golang-migrate` подойдёт).

**Задача 1.3 — event-collector-uptime**
Сервис, который по расписанию (например каждые 30 сек) пингует список URL
из `sources` (HTTP GET, таймаут, ожидаемый статус), при смене состояния
(up→down / down→up) пишет строку в `events`.

**Задача 1.4 — api: CRUD источников + история событий**
REST-эндпоинты: добавить/удалить/список источников, получить последние N
событий с пагинацией.

**Задача 1.5 — ws-gateway**
WS-сервер на `nhooyr.io/websocket`. Подписка через Postgres `LISTEN/NOTIFY`
на новые строки в `events`, рассылка подключённым клиентам как JSON.

**Задача 1.6 — UI (минимальный)**
Один HTML-файл + чистый JS (без сборщиков и фреймворков): список источников
(добавить URL для мониторинга), живая лента событий через WS, статус
online/offline индикатором.

Критерий готовности спринта: открыл браузер на localhost, добавил URL для
мониторинга, увидел живое обновление статуса без перезагрузки страницы.

## Спринт 2: больше источников, шина событий, приоритеты

**Задача 2.1 — переход на NATS**
Поднять NATS в docker-compose. `event-collector-uptime` публикует в NATS
вместо прямой записи + LISTEN/NOTIFY. Новый маленький сервис-подписчик
пишет из NATS в Postgres. `ws-gateway` подписывается на NATS вместо
Postgres-нотификаций.

**Задача 2.2 — event-collector-rss**
Использовать `mmcdole/gofeed`. Источник типа `rss` в конфиге хранит URL
фида, периодически проверяет новые записи, публикует события в NATS.

**Задача 2.3 — event-collector-github**
Через GitHub API (personal access token в конфиге источника) — новые
релизы/уведомления по репозиторию.

**Задача 2.4 — event-collector-price**
Курс валюты/крипты через любой публичный API, публикует событие при
пересечении заданного порога (порог хранится в config источника).

**Задача 2.5 — priority и read_at в модели**
Добавить `priority` (low/normal/high) на событие при публикации каждым
коллектором. UI: фильтр по источнику и приоритету, кнопка "отметить
прочитанным".

**Задача 2.6 — mute источника**
Флаг `muted_until` на источнике, простой контрол в UI ("замьютить на 1ч /
до утра / насовсем"), доставка проверяет флаг перед публикацией в WS.

Критерий готовности: минимум 4 разных типа источников работают
одновременно, лента фильтруется, можно замьютить шумный источник.

## Спринт 3: offline-доставка и мобильный клиент

**Задача 3.1 — таблица notifications и delivery-service**
Таблица `notifications` (event_id, status: pending/delivered/pushed,
attempts, created_at). Новый сервис слушает NATS, создаёт запись,
пытается WS-доставку, ждёт ack-сообщение от клиента по WS.

**Задача 3.2 — FCM интеграция**
Firebase-проект создать вручную (агенту описать шаги, не делать за
пользователя — нужны реальные учётные данные). `firebase.google.com/go/v4`
в delivery-service: если ack не пришёл за таймаут — push через FCM,
статус → `pushed`.

**Задача 3.3 — дедупликация/агрегация**
Окно агрегации (например 60 сек) по паре (source_id, similar title):
повторяющиеся события схлопываются в одно с счётчиком вместо N уведомлений.

**Задача 3.4 — Kotlin/Compose Multiplatform клиент, каркас**
Экран ленты (список событий с приоритетом), экран источников
(добавить/удалить/замьютить), подключение к WS API.

**Задача 3.5 — обработка push на клиенте**
Регистрация FCM-токена на устройстве, отправка токена в `api`, обработка
входящего push, открытие соответствующего события при тапе.

Критерий готовности: закрыл приложение на телефоне → событие всё равно
прилетело как системный пуш, без задержки. Свернул на 5 минут → при
возврате лента актуальна.

## Спринт 4: деплой, самодиагностика, полировка

**Задача 4.1 — перенос на VPS**
docker-compose на арендованном сервере (агент не покупает сервер сам —
пользователь заранее подготавливает VPS и даёт SSH-доступ/креды).
Caddy как reverse proxy, автоматический TLS через Let's Encrypt, домен
указан на IP сервера.

**Задача 4.2 — авторизация**
Минимальный токен/пароль на `api` и WS-соединение — теперь сервис смотрит
в интернет, анонимный доступ недопустим.

**Задача 4.3 — внешний watchdog**
Health-эндпоинт `api`, задокументировать (не обязательно кодом) как
подключить внешний бесплатный uptime-чекер на этот эндпоинт — сам хаб не
может мониторить собственное падение.

**Задача 4.4 — метрики и Grafana**
Экспорт Prometheus-метрик (кол-во событий/час, задержка доставки WS vs
push, соотношение доставок по каналам), VictoriaMetrics + Grafana в
docker-compose, готовый dashboard.

**Задача 4.5 — UI-полировка**
Тёмная тема, сортировка ленты, простой текстовый поиск по истории
уведомлений.

**Задача 4.6 — интеграционные тесты и чистка**
`testcontainers-go` для ключевых сценариев (событие → WS-доставка,
событие → push при офлайне, дедупликация). README с инструкцией по
локальному запуску и по деплою. Финальная чистка docker-compose и env-файлов.

Критерий готовности: сервис работает на VPS 24/7, доступен по домену,
есть базовые метрики в Grafana, есть тесты на критичный путь доставки.

## Как агенту двигаться по этому файлу

1. Выполнять задачи строго по порядку номеров.
2. Не начинать задачу из следующего спринта, пока текущий не завершён и
   не подтверждён пользователем.
3. Если задача оказывается больше, чем ожидалось — сообщить об этом и
   предложить разбить на подзадачи, а не молча расширять скоуп.
4. Если для задачи не хватает внешних данных (API-ключ, аккаунт, домен) —
   явно попросить их у пользователя, не придумывать заглушки, которые
   потом придётся выковыривать.
