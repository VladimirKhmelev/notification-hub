# Project Brief: Personal Notification Hub

## Что строим

Self-hosted личный хаб уведомлений. Разные источники событий (аптайм серверов, RSS,
GitHub-уведомления, курсы валют/крипты и т.д.) публикуют события в единый пайплайн.
Пользователь (я один, multi-tenancy не нужен на этом этапе — весь проект single-tenant)
видит их живой лентой в вебе, а также получает push-уведомления на телефон, когда
не смотрит в приложение активно.

Цель — не мокап, а реально рабочая штука, которой я пользуюсь каждый день.
UI должен появиться уже в Спринте 1, дальше улучшается постепенно —
не откладывать интерфейс "на потом".

## Ключевой принцип доставки (важно, не отклоняться)

Два канала:
1. **WebSocket** — мгновенная доставка, пока клиент онлайн.
2. **FCM push** — фолбэк, когда клиент не подтвердил получение по WS в течение
   короткого таймаута (2–3 сек).

Событие → пишется в БД со статусом `pending` → доставка пытается через WS →
если пришёл ack от клиента — статус `delivered`, push НЕ шлём →
если ack не пришёл за таймаут — шлём push, статус `pushed`.
Цель: не дублировать уведомление на два канала одновременно.

## Технологический стек

- **Язык:** Go (бэкенд), Kotlin/Compose Multiplatform (мобильный клиент, начиная со Спринта 3)
- **БД:** PostgreSQL
- **Шина событий:** сначала Postgres `LISTEN/NOTIFY` (Спринт 1), с Спринта 2 — NATS
- **WebSocket:** `nhooyr.io/websocket`
- **RSS:** `mmcdole/gofeed`
- **Push:** `firebase.google.com/go/v4` (FCM HTTP v1 API)
- **Reverse proxy / TLS:** Caddy (только в Спринте 4, для VPS)
- **Мониторинг:** Prometheus-совместимые метрики + VictoriaMetrics + Grafana (Спринт 4)
- **Тесты:** стандартный `testing` + `testcontainers-go` для интеграционных
- **Инфраструктура:** Docker Compose, никакого Kubernetes

## Архитектура сервисов (растёт по спринтам, не создавать всё сразу)

- `api` — REST для управления источниками, историей событий, аутентификацией
- `ws-gateway` — держит WS-соединения, транслирует события подписчикам
- `event-collector-*` — по одному сервису на тип источника (uptime, rss, github, price)
- `delivery-service` — логика WS-ack / FCM fallback (появляется в Спринте 3)
- UI — простой SPA (обычный HTML/JS без тяжёлого фреймворка на вебе), Kotlin/Compose клиент отдельно

## Конвенции разработки

- Двигаться маленькими вертикальными срезами: сквозной работающий кусок (backend + UI),
  а не "сначала весь бэкенд, потом весь фронт".
  ного/ы
- После каждой значимой задачи — рабочий коммит с понятным сообщением (conventional commits:
  `feat:`, `fix:`, `chore:`, `test:`).
- Каждый сервис — отдельный `go module` внутри монорепы ИЛИ единый модуль с
  пакетами `internal/<service>` — на усмотрение агента, но выбранный подход
  описать в `ARCHITECTURE.md` и не менять посреди спринта.
- docker-compose должен всегда быть в рабочем состоянии на конец каждой задачи
  (`make up` поднимает всё, что уже реализовано).
- Не добавлять зависимости/сервисы, не предусмотренные текущим спринтом,
  даже если "пригодится в будущем" — переусложнение чуждо этому проекту
  (см. заметку ниже).

## Явное анти-требование

Прошлый похожий проект получился переинженеренным для своего масштаба
(слишком много сервисов ради простой задачи поиска по комиксам).
Здесь это неприемлемо: сложность должна быть оправдана конкретной задачей
спринта. Если для задачи достаточно одной функции без нового сервиса —
делать так.
